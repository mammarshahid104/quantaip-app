# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```sh
npm start          # Start Metro bundler
npm run android    # Build and run on Android
npm run ios        # Build and run on iOS
npm run lint       # Run ESLint
npm test           # Run Jest tests
npx jest __tests__/App.test.tsx   # Run a single test file
```

Node >= 22.11.0 is required. For iOS first-time setup: `bundle install && bundle exec pod install`.

Metro for device testing runs with `--host 0.0.0.0 --port 8081`; connect Android devices via ADB wireless debugging.

## Architecture

**QUANTAIP Education OS** — a React Native school management app backed by Firebase (Auth + Firestore).

### Design System

Dark purple (`#1e1b4b`, `#7c3aed`) on white with Poppins as the primary font. Background is `#faf5ff`, accent purple `#a78bfa`, cards use `#ffffff` with `#ede9fe` borders.

### Hard Rules

- **Never add `firebase-admin` to app dependencies.** It belongs only in the separate `admin-scripts` folder (server-side tooling). The app uses `@react-native-firebase/*` exclusively.
- **Attendance reads must always use `attendanceMap` on the student doc.** Never query the per-day `attendance/{dateKey}/{className}` sub-collection from client screens — that collection is write-only from the teacher side and exists only for audit. The dual-write to `attendanceMap` is the intended read path.

### Authentication & Role Detection

`App.tsx` is the entry point. Login parses a structured user ID (e.g. `GHS-001-TCH-0001`) to detect the role from the `-ADM-`, `-TCH-`, `-STU-`, or `-PAR-` segment, then authenticates via Firebase Auth using the email convention `<id-lowercase>@quantaip.edu.pk`. After login, the navigator resets to the role-specific screen.

### Navigation

A flat `NativeStack` navigator with five screens: `Login`, `Admin`, `Teacher`, `Student`, `Parent`. No nested navigators. All role screens receive `navigation` as a prop and call `navigation.reset({index: 0, routes: [{name: 'Login'}]})` on sign-out to clear the stack.

### Firestore Data Model

All data is scoped under `schools/{SCHOOL_CODE}/`. The school code is the single constant in `config.ts` and is imported into every screen.

Key sub-collections:
- `students/{studentId}` — profile + `attendanceMap: {YYYY-MM-DD: 'P'|'A'|'L'}` (dual-written for fast reads)
- `teachers/{teacherId}` — profile + `classesAssigned[]` + `classIncharge` (set in `ClassesScreen`)
- `attendance/{dateKey}/{className}/{studentId}` — primary attendance records
- `timetable/{className}` — weekly schedule document `{Monday: [{period, subject, teacher, time}], ...}`
- `marks/{testId}` — test metadata; `marks/{testId}/students/{studentId}` — per-student marks
- `results/{resultId}/students/{studentId}` — compiled ranked results (generated by Admin)
- `fees/{month}/students/{studentId}` — fee payment records
- `homework/{className}` — single doc with `items[]` array (capped at 50, newest first)
- `classes/{className}` — sections array + classIncharge teacher ID
- `parents/{parentId}` — linked to `studentId`

### Secondary Firebase App (Account Creation)

`AdminScreen.tsx` uses a secondary Firebase app instance named `USER_CREATOR` to create new Auth accounts without displacing the admin's own session. The helper `createAuthAccount` re-uses or lazily initializes this secondary app. This is a deliberate workaround for Firebase's behavior of switching the active session on `createUserWithEmailAndPassword`.

### Screen Structure

Each role screen is self-contained with its own tab navigation (rendered as `ScrollView` of `TouchableOpacity` chips, not React Navigation tabs). All styles are defined via `StyleSheet.create` at the bottom of each file.

- **AdminScreen** — tabs: Dashboard, Classes (→ ClassesScreen), Students, Teachers, Fee (→ FeeScreen), Results, Timetable, Import. Handles bulk Excel import via `@react-native-documents/picker` + `xlsx`.
- **TeacherScreen** — tabs: Attendance (only for class incharges), Marks (3-step wizard), Homework, Timetable, My Classes. Own-period highlighting in timetable.
- **StudentScreen** — tabs: Overview, Attendance, Homework, Grades, Results, Timetable. Reads attendance from `attendanceMap` on the student doc.
- **ParentScreen** — mirrors StudentScreen for the linked child.
- **ClassesScreen** — embedded in AdminScreen's Classes tab; manages sections and class incharge assignment.
- **FeeScreen** — embedded in AdminScreen's Fee tab; fee structure per class + per-student fee type overrides.

### ID Format

IDs follow the pattern `{SCHOOL_CODE}-{ROLE}-{NNNN}` (admin uses 3 digits, others 4). `formatId` in `App.tsx` auto-formats as the user types by stripping non-alphanumerics and inserting hyphens. `generateId` in `AdminScreen.tsx` generates sequential IDs based on current collection size.

### Class Hierarchy

`CLASS_HIERARCHY` (duplicated in several screens) defines four categories: Early Education (Nursery, Prep, KG), Primary (Grades 1–5), Middle (Grades 6–8), Secondary (Grades 9–12). This constant is the source of truth for class selectors throughout the app.

### Timetable Structure

Each class document in `timetable/` has keys for each day. Each day is an array of slot objects `{period, subject, teacher, time}`. `period: 0` is the break slot. Friday uses shorter timings (30 min periods vs 40 min). Admin can copy one day's subjects/teachers to all other days while preserving each day's correct timings.
